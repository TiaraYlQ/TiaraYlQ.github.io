NorthCombat 模组开发

在 Minecraft 里复刻出接近现代战术射击体验的模组——NorthCombat，这个想法诞生已经有一段时间了。我目前把整个模组的交互、功能模块、网络协议和开发节奏梳理成了一份完整的需求文档。

# NorthCombat 模组开发日志：从零构建一款战术FPS模组的设计蓝图

距离在《我的世界》里复刻出接近现代战术射击体验的模组——**NorthCombat**，这个想法诞生已经有一段时间了。经过几轮内部讨论与技术验证，我们最终把整个模组的交互哲学、功能模块、网络协议和开发节奏梳理成了一份完整的需求文档。这篇博客既是归档记录，也是方便团队成员随时对齐认知的“活文档”。

1. 核心理念：让指令从玩家视野中彻底消失

传统模组，尤其是多人竞技模组，往往要求玩家记忆大量指令/join/create/ready……这在快节奏对战中非常割裂。NorthCombat 的设计第一原则就是“零指令”。

玩家的唯一交互入口是一件物品——战术终端。它固定在物品栏的第9格，无法丢弃、无法移动，死亡后自动恢复。右键打开后，一个基于 MCEF 的内置浏览器会加载我们的平台官网。后续所有操作——签到、建房、匹配、查看数据——都在这个网页中点击完成。网页通过 JavaScript 桥接调用模组能力，模组再通过网络包与服务端交互。

对管理员（OP）来说，后台指令依然保留，但对普通玩家完全透明。

2. 官网功能

网页端承担了几乎全部“大厅系统”的职能，我们把它划分为9大模块：

| 签到 | 每日打卡，连续签到递增奖励 |

| 活动 | 限时/节日任务，一键参与，自动结算 |

| 建房 | 自定义模式、地图、密码、人数、排位开关与规则参数 |

| 排位 | 段位、ELO、赛季数据、历史对局详情 |

| 背包 | 搜打撤装备仓库、武器配置、保险箱管理 |

| 好友 | 增删好友、邀请、在线状态 |

| 排行榜 | 全服/赛季排名，多维度排行 |

| 商店 | 皮肤、通行证、虚拟货币充值 |

每个模块都对应独立的网页面板，通过左侧导航栏切换，右侧动态渲染内容，实现类似桌面应用的操作体验。

3. 建房流程：从表单到房间实例

建房是完全在网页内闭环的，流程如下：

1. 玩家点击“创建房间”，弹出面板；

2. 选择模式（死斗/TDM/竞技/搜打撤）、地图（从服务端实时拉取列表）、最大人数、可选密码、是否排位，以及自定义规则（回合时间、击杀上限等）；

3. 点击“创建”按钮，网页调用 NorthCombat.createRoom(configJson)；

4. 模组发送 C_CREATE_ROOM 包到服务端，服务端验证权限后创建房间实例；

5. 网页自动跳转至“我的房间”面板，展示房间号、当前玩家列表、房主专属的“开始游戏”和“邀请好友”按钮，以及每位玩家的准备状态。

这种设计让建房与匹配在体验上完全统一，无需离开终端屏幕。

4. 排位与对局记录：数据可追溯

排位信息页不再只是一张静态图，而是动态仪表板：

- 当前段位图标与名称；

- ELO 进度条（显示当前分数与升级门槛）；

- 赛季关键指标：场次、胜率、KD、ADR、爆头率等等；

- 最近10场对局缩略列表（可展开完整详情）；

- 段位历史折线图；

- 赛季剩余倒计时与奖励预览。

对局记录则更偏向复盘分析：

- 列表视图：时间、模式、地图、结果、得分、时长；

- 详情视图：双方完整玩家列表、个人数据（击杀/死亡/助攻/伤害/经济）、击杀位置热区图（计划用 Canvas 绘制）、下载回放按钮；

- 若玩家正在对局中，该页面会实时推送比分、存活状态、回合历史和经济曲线（竞技模式专属），为观众或教练提供数据支持。

5. 网络包协议：客户端与服务端的共同语言

为了低耦合与可维护性，我们严格区分包的方向与职责。

客户端 → 服务端C_ 系列）：

- C_MATCH_REQUEST – 快速匹配

- C_CREATE_ROOM – 创建房间（附带完整 JSON 配置）

- C_JOIN_ROOM / C_LEAVE_ROOM – 房间进出

- C_KICK_PLAYER / C_START_GAME – 房主权限

- C_PLAYER_READY – 准备/取消准备

- C_DAILY_CHECKIN / C_CLAIM_REWARD / C_CLAIM_ACTIVITY – 活动与签到

- C_INVITE_FRIEND / C_ACCEPT_INVITE – 好友邀请

- C_SEND_CHAT – 游戏内文本通信

- C_UPDATE_SETTING – 客户端设置同步

- C_BUY_ITEM / C_PLANT_BOMB / C_DEFUSE_BOMB – 竞技模式核心机制

- C_EXTRACT – 搜打撤撤离

服务端 → 客户端S_ 系列）

- S_ROOM_CREATED / S_ROOM_UPDATED / S_ROOM_LIST – 房间状态同步

- S_MATCH_FOUND – 匹配成功，分配队伍

- S_PLAYER_JOINED_ROOM / S_PLAYER_LEFT_ROOM – 玩家进出通知

- S_ALL_READY / S_GAME_START – 倒计时与开赛

- S_GAME_STATE – 对局全局状态

- S_SYNC_PLAYER_DATA – 经济、弹药、生命等关键数据

- S_DEATH_EVENT / S_KILLCAM_DATA – 击杀与死亡回放

- S_SPAWN_PROTECTION – 出生保护

- S_ROUND_START / S_ROUND_END / S_MATCH_END – 回合生命周期

- S_BUY_MENU_OPEN – 购买阶段触发

- S_BOMB_PLANTED / S_BOMB_DEFUSED – 炸弹状态

- S_EXTRACTION_ZONE – 撤离点信息

- S_CHECKIN_RESULT / S_CLAIM_RESULT – 活动结算

- S_RANK_DATA / S_LEADERBOARD / S_FRIEND_LIST / S_INVITE_RECEIVED – 社交与数据

- S_SETTING_SYNC / S_ERROR – 同步与错误处理

每个网络包都会实现为独立文件，方便后续扩展与多人协作。

6. 网页端 UI 布局与桥接 API

界面结构遵循经典的仪表板式设计：

- 顶部栏：显示玩家头像、昵称、段位勋章、虚拟货币余额；

- 左侧导航（图标+文字）：首页、签到、活动中心、快速匹配、创建房间、排位赛、对局记录、我的背包、好友、排行榜、设置、关闭终端；

- 右侧内容区：根据导航项动态切换，保留面包屑导航。

7. 管理员工具与地图管理

虽然普通玩家完全无感，但服务器维护离不开 OP 指令。我们保留了6条核心命令：

- /tc stop – 强制结束当前对局

- /tc setlobby <radius> – 设定大厅保护区半径

- /tc map create <id> / delete <id> – 动态管理地图

- /tc map wand <id> <team> – 使用选区工具设置队伍出生点

- /tc reload – 热重载配置文件

- /tc debug <on/off> – 开启或关闭调试日志输出

这些命令只在后台执行，不会出现在任何面向玩家的界面中。

8. 分阶段开发计划（共8个阶段）

为了控制复杂度，我们按依赖关系拆解成8个冲刺阶段：

1. 基础框架：主类、网络包注册机制、配置文件加载、管理员命令骨架。

2. 核心系统：GameState 状态机、MatchManager 与 MatchInstance 生命周期、TeamManager / Team 数据结构。

3. 战术终端：特殊物品实现、第9格锁定逻辑（包含防丢弃、防移动、死亡恢复）、全面的物品保护拦截。

4. 浏览器界面：MCEF 屏幕渲染、JS 桥接对象 NorthCombat 的方法注册与回调推送。

5. 全部网络包：逐一实现所有 C_ 和 S_ 包，每个包独立文件，编写单元测试。

6. 平台通信：HTTP 客户端封装，对接外部平台 API（签到奖励、排位数据拉取、商店查询等）。

7. TACZ 集成：兼容 Timeless and Classics Guns 模组，构建虚拟装备系统，支撑搜打撤模式的进/出局装备逻辑。

8. 游戏模式：抽象模式接口，依次实现死斗、TDM、竞技（含经济系统与炸弹逻辑）、搜打撤四种模式。
